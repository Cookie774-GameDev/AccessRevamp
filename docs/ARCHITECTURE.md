# Architecture

**Status:** Mixed — `IMPLEMENTED` platform foundation, `PLANNED` production-rebuild modules, `EXTERNALLY BLOCKED` remote-state verification, and `LAUNCH-ONLY` activation.

**Owner:** Engineering. Payment, database, operations, and security owners review changes to their boundaries.

**Authority:** [approved rebuild specification](superpowers/specs/2026-07-18-accessrevamp-production-rebuild-design.md) and the [foundation/catalog plan](superpowers/plans/2026-07-18-accessrevamp-foundation-catalog.md).

## Retained platform

AccessRevamp remains a Vite single-page application deployed with Netlify Functions, Supabase Auth/Postgres, and Stripe-hosted Checkout. The decision is recorded in `docs/adr/0001-retain-vite-netlify-supabase-stripe.md`. A framework, CMS, WebGL, payment-provider, or database migration is outside the approved rebuild.

The browser owns rendering, navigation, accessible interaction state, and safe public catalog projection. Netlify Functions own authenticated mutations, strict request validation, server credentials, Stripe selection, webhook verification, and service-role database access. Supabase owns identity, durable relational state, transactions, RLS, and narrowly granted RPCs. Stripe owns hosted payment collection and test-mode events.

## Authority boundaries

1. `src/config/tier-catalog.js` will define non-secret tier keys, ranks, names, cents, currency, descriptions, deliverables, and allowed transitions.
2. `netlify/functions/_shared/stripe-catalog.mjs` will map those keys to server-only environment variable names and test Price IDs.
3. The browser may send only a target tier and random request ID. It never sends an accepted amount, credit, Price ID, user ID, or entitlement.
4. A transaction or security-definer RPC locks the customer entitlement and creates the exact upgrade reservation.
5. Checkout uses the reservation-selected server Price ID and the nine-field metadata contract.
6. The webhook re-retrieves and verifies Stripe data before atomically updating event, order, reservation, entitlement, project, and audit state.

Google Drive is an internal operator-context source only. No browser route reads Drive, no Drive credential enters the bundle, and no public asset is generated by copying prospect records or secrets from Drive. Any future operator import must be explicit, minimized, logged, server-side, and reviewed.

## Application structure

- `src/app/router.js` matches static and parameterized routes and runs route-local cleanup.
- `src/app/metadata.js` owns route metadata without embedding sensitive tokens.
- `src/pages/` owns public, account, operator, preview, policy, and demo page modules.
- `src/components/` owns reusable accessible controls and editorial sections.
- `src/demos/` keeps Greenline, Firejar, and Clearflow state and styles route-isolated.
- `netlify/functions/` is the same-origin HTTP boundary.
- `netlify/functions/_shared/` contains reusable server validation, auth, catalog, logging, and response helpers.
- `supabase/migrations/` contains forward-only schema changes in filename order.
- `docs/evidence/` retains secret-safe verification artifacts.

## Reliability model

Primary content and actions render without optional media, animation, analytics, map, weather, or authentication. Each route owns event listeners, timers, animation frames, observers, and abort controllers and disposes them on navigation. Requests have bounded bodies and timeouts. Loading, empty, partial, expired, unauthorized, rate-limited, unavailable, and retry states are designed rather than collapsed into generic failure.

Reservations expire after 30 minutes. Reused request IDs are idempotent. Webhook event IDs are deduplicated without discarding a recorded-but-unprocessed retry. Refund dependencies prevent a base refund from silently preserving downstream credit.

## Security shape

RLS applies to every exposed customer table. Internal research, suppression, queue, Stripe event, reservation, dependency, and audit records remain server/operator only. Security-definer functions use a fixed safe `search_path`, least-privilege ownership, explicit grants, and revoked public execution. Logs use allowlisted fields and never serialize secrets, raw preview tokens, full Stripe objects, or free-form prospect/customer content.

## Delivery states

### IMPLEMENTED

Vite, Netlify Functions, Supabase client/server separation, Stripe signature verification, same-origin checks, RLS-aware customer tables, route cleanup, baseline tooling, and the linked worktree are present locally.

### PLANNED

The canonical catalog, declarative router, entitlement RPC, reservation flow, refund reconciliation, operator projections, demos, analytics adapter, and recovery tooling are specified in the implementation plans.

### EXTERNALLY BLOCKED

The connected Supabase project, Stripe test catalog, webhook endpoint, and Netlify preview have not been proven from this worktree. Remote schema, policies, products, and runtime headers require connector or owned staging evidence.

### LAUNCH-ONLY

Production environment binding, live Price creation, production migrations, domain cutover, secret rotation, monitoring enablement, and outreach activation require independent approval and a completed rollback rehearsal.
